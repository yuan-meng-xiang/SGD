Lesson 0001

Codex 不是代码生成按钮

这一课只练一个能力：你能不能把 Codex 的定位讲清楚，而不是把它讲成“又一个会写代码的聊天框”。

主教材：`Codex橙皮书.pdf` 第 5-15 页。教材标注为非官方指南，v0.1.0，最后校验于 2026-06-22；涉及安装入口、额度、价格、模型名称时要再查官方当前资料。

先抓住角色变化

讲 Codex 的第一句话，不要从“它能写什么代码”开始，而要从“它在软件工程里扮演什么角色”开始。

Codex 是可以进入项目现场、推进软件工程任务到可 review 状态的 AI coding agent。

这句话里有三个关键词：项目现场、工程任务、可 review。它们把 Codex 和普通聊天式写代码工具区分开。

三个关键词

项目现场

Codex 不是只看你复制的一段代码。它可以围绕当前项目读文件、理解上下文、搜索代码、运行命令。

工程任务

适合它的不是“随便做个大项目”，而是边界清楚的小任务：修 bug、加功能、补测试、写文档、整理 PR。

可 review

Codex 的产物不应该只停留在回答里，而应该能被你看 diff、跑测试、检查风险，再决定是否接受。

给别人讲时的对比

  • ChatGPT：更像顾问，帮你想清楚。
  • Cursor：更像 AI 编辑器，陪你改代码。
  • Codex：更像工程 Agent，帮你推进任务。

这不是说谁替代谁，而是讲清楚使用场景：想问题、写局部、交付任务，是三个不同动作。

什么时候适合用 Codex

你可以用六个判断词：目标明确、范围可控、上下文清楚、结果能验证、失败能回滚、风险可接受。

反过来，生产数据库、真实用户数据、支付核心逻辑、安全权限核心模块、你自己也无法验收的任务，都不适合直接交给它独立处理。

参考答案：ChatGPT 更像一个顾问，主要帮你分析问题、解释代码、生成思路；你需要把答案自己搬回项目里执行。Codex 更像一个能进入项目现场的工程 Agent，它可以读文件、改代码、跑命令、看 diff，把一个边界清楚的软件任务推进到可 review 的状态。
