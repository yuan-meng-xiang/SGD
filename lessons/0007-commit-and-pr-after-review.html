Lesson 0007

验收之后提交与 PR

这一课只练一个能力：当 Codex 的改动通过验收后，你能不能把它变成一次清楚、可回退、可 review 的交付。

主教材：`Codex橙皮书.pdf` 第 132-134 页、第 169-176 页。补充来源：Pro Git: What is Git?、Pro Git: Interactive Staging、GitHub: About pull requests、Conventional Commits。

验收后还没结束

第六课解决的是“能不能接受这次改动”。第七课解决的是“接受之后怎么保存和交付”。如果只让代码留在工作区里，过几天你很难说清楚这次到底改了什么、为什么改、怎么验证过。

stage 是选择，commit 是存档，PR 是申请，merge 是合入。

这句话能把 Git 和 Codex 的关系讲清楚：Codex 负责帮你完成任务，Git 负责记录变化，GitHub/PR 负责把变化交给人 review。

四个动作不要混

动作 它回答什么问题 给新手的讲法
stage 哪些改动准备放进这次提交？ 只把通过验收、属于同一件事的改动放进去。
commit 这次保存的版本是什么？ 给项目拍一张可回退的快照，并写清楚改了什么。
push 本地提交是否同步到远程？ 把本地记录上传到 GitHub 等远程仓库。
PR 是否准备把改动合进主线？ 提交一份“请检查后合并”的修改申请。

先 stage，再 commit

不要把“所有当前改动”自动等同于“这次应该提交的改动”。第六课里你已经学会了看 diff；现在要把这个判断落实到 stage。

适合放进同一个 commit

同一个目标、同一组文件、同一套验证结果。比如“给登录页增加忘记密码链接”。

应该拆开的改动

功能修改、全局样式、依赖升级、无关重构、文档补充混在一起时，通常要拆开。

Pro Git 把 staging area 描述成下一次 commit 的准备区。教学时可以更口语化：stage 就是在说“这次我要保存哪些改动”。

检索练习 1

在 Codex 改完并通过验收后，stage 最接近哪个动作？

commit message 要说清楚范围

好的 commit message 不要夸大，也不要空泛。它应该让未来的你一眼看懂：这次主要改了什么。

类型 适合场景 例子
feat 新增用户可见能力。 `feat: add reset link`
fix 修复错误行为。 `fix: restore login redirect`
docs 只改文档。 `docs: update setup guide`
refactor 不改变外部行为的结构调整。 `refactor: simplify form state`

如果使用 Conventional Commits，常见格式是 `type: description`。但格式不是重点，重点是范围真实、描述具体、不要把“修一个链接”写成“重构登录系统”。

检索练习 2

任务是“给登录页新增忘记密码链接”。哪个 commit message 更合适？

PR 不是走形式

GitHub 文档把 pull request 解释为合并代码改动的提案，可以在合并前讨论和 review。给新手讲时，可以说：PR 是把“代码改动 + 验证证据 + 风险说明”打包交给 reviewer。

做了什么 为什么做 怎么验证 风险在哪

如果没有团队 reviewer，也可以把 PR 当成给未来自己的交付记录。尤其是 Codex 参与的改动，PR 描述要写清楚哪些是自动生成、哪些经过人工验收。

给 Codex 的交付提示词

请进入提交与 PR 准备阶段，不要继续新增功能。请基于当前 diff 和测试结果输出：1. 建议 stage 哪些文件；2. 一个不夸大范围的 commit message；3. PR 描述，包括本次修改、修改原因、测试结果、风险说明、reviewer 需要重点看的地方；4. 是否还有不应该提交的改动。

今天要能说出口

请用 4-6 句话回答：为什么 Codex 改完并通过验收后，还需要 stage、commit 和 PR？stage、commit、PR、merge 分别解决什么问题？为什么不能把所有改动直接一键提交或自动合并？

写给新手听的版本

要包含三个点：stage 是选择；commit 是可回退记录；PR 是合并前的审查申请。

参考答案：Codex 的改动通过验收后，还需要用 Git 把它变成清楚、可回退、可 review 的交付。stage 是选择哪些改动进入这次提交，避免把无关文件也保存进去；commit 是给这次通过验收的改动做一次版本存档，并用 message 说明范围；PR 是把改动、原因、测试和风险交给自己或团队再检查；merge 才是把它合进主线。不能把所有改动一键提交或自动合并，因为 Codex 可能留下无关重构、临时文件、配置变化或未验证风险。好的流程是：验收通过，只 stage 相关改动，写清楚 commit，再用 PR 做最后一层 review。
