管好 Codex App 的任务空间

这一课只练一个能力：你能不能给新手讲清楚 Codex App 里项目、thread、搜索、等待批准和归档各自解决什么问题。

主教材：`Codex橙皮书.pdf` 第 31-41 页。这节课只讲任务空间管理；设置、权限、沙盒、模型和计划模式放到下一课。

App 不是一个大聊天框

很多新手会把 Codex App 当成“带文件能力的 ChatGPT”。但真正开始做项目后，更重要的是理解它的任务空间：项目决定 Codex 能看哪里，thread 决定当前任务边界，归档和搜索负责让任务列表不失控。

项目决定 Codex 能看哪里，thread 决定这次只做什么，归档让任务列表保持干净。

先讲三栏布局

给别人演示 Codex App 时，可以先用三栏建立空间感：

区域 它负责什么 新手怎么理解
左侧 项目列表、thread 列表、搜索和入口导航。 这里负责“我在哪个项目、哪个任务里”。
中间 对话和任务窗口。 这里负责“我要 Codex 做什么、它准备怎么做”。
右侧 多功能区域，例如文件、diff、浏览器和注释。 这里负责“我怎么检查和辅助它完成任务”。

项目对话和非项目对话

新建对话时，一个关键判断是：这次是否需要 Codex 进入某个代码项目。

使用项目

适合读项目、改代码、修 bug、生成 README、检查 diff。因为 Codex 需要看到项目文件，才能做工程任务。

不使用项目

适合问概念、讨论方案、准备讲稿、比较工具。它更像普通问答，不应该期待它直接修改本地文件。

所以你可以这样教：只要任务会落到文件里，就选项目；只是想清楚问题，可以不使用项目。

thread 是任务边界

一个项目里可以有多个 thread，每个 thread 专门处理一个具体任务。教材里举的例子很好：做首页、修按钮、优化移动端、写 README，最好是不同 thread。

一个明确任务 = 一个 thread。不要把“做页面、修 bug、改样式、写文档”全塞进一个对话。

这样做有两个好处：Codex 不容易上下文混乱，你也更容易回看每个任务到底改了什么。

但多 thread 不是万能的。如果任务分解没做好，多个 thread 仍然可能互相干扰：比如两个 thread 都在改同一批文件，或者一个 thread 做布局、另一个 thread 同时改同一区域的样式。更稳的原则是：一个 thread 对应一个明确目标、一个主要区域、一个可验收结果；如果新任务依赖旧任务，要明确告诉 Codex “基于当前版本继续”。

搜索、等待批准和归档

  • 搜索：thread 多了以后，用关键词找回历史任务，不要靠记忆翻列表。
  • 等待批准：Codex 需要权限时会停下来。先看清它要做什么，再点击允许。
  • 归档：完成或暂时不用的 thread 收起来。归档不是删除代码，也不是合并代码，只是整理任务列表。

这三个功能都不是“高级功能”，而是保持任务空间清晰的基本卫生习惯。

写给新手听的版本

要包含三个点：项目决定工作范围；thread 是任务对话；搜索、等待批准、归档帮助管理任务。

参考答案：项目可以理解成 Codex 要进入的代码文件夹或仓库，它决定 Codex 能看到哪些文件、在哪个范围里工作。thread 是这个项目里的一个具体任务对话，一个 thread 最好只处理一个明确任务，比如做首页、修按钮、优化移动端要分开。这样 Codex 的上下文更清楚，你也更容易检查每个任务对应的 diff 和结果。但多 thread 不能替代任务分解，如果多个 thread 改同一批文件或任务边界重叠，仍然可能互相干扰。任务变多后，可以用搜索找回历史 thread；遇到等待批准时先看清请求；完成或暂时不用的 thread 可以归档，让任务列表保持干净。
