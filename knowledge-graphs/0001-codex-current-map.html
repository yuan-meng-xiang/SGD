Knowledge Graph 0001

Codex 当前知识图谱快照

这是阶段地图，不是最终百科。它只记录当前已经学过、正在引入、以及后续要接上的 Codex 主干。

Snapshot date: 2026-06-29. Built from lessons 0001-0005 and learning records 0001-0004.

薄框架

1. 定位

Codex 是能进入项目现场、推进工程任务到可 review 状态的 AI coding agent。

Status: learned. Evidence: LR-0001, Lesson 0001.

2. 入口

App、CLI、IDE Extension、Web/Cloud 不是难易等级，而是适配不同工作流的入口。

Status: learned. Evidence: LR-0002, Lesson 0002.

3. 工作流

选项目、开 thread、给小任务、看计划、看执行、查 review pane、验 diff。

Status: learned. Evidence: LR-0003, Lesson 0003.

4. 上下文

Codex 的质量取决于项目目录、任务边界、thread 范围、已有代码和可验证目标。

Status: learned. Evidence: LR-0004, Lesson 0004.

5. 执行与验证

Codex 的产物要通过 diff、测试、运行结果和人工判断来验收，而不是只看文字总结。

Status: learned. Evidence: LR-0003, Lesson 0001 and 0003.

6. 扩展能力

Skills、plugins、MCP、automations 会让 Codex 变成更贴近项目和个人工作流的 agent。

Status: planned. Needs later track.

7. 安全边界

安全边界来自任务边界、沙盒、审批和 review 的组合；适合交给 Codex 的任务应目标明确、范围可控、结果能验证、失败能回滚、风险可接受。

Status: introduced. Evidence: Lesson 0001, 0003 and 0005.

关键关系

  • 定位 -> 工作流：因为 Codex 是工程 agent，所以教学重点不是聊天技巧，而是任务闭环。
  • 入口 -> 工作流：入口决定在哪里运行同一套闭环；新手通常先用 App 看完整过程。
  • 上下文 -> 执行与验证：上下文决定 Codex 能做什么，验证决定结果能不能被接受。
  • 项目 -> thread：项目提供文件级上下文，thread 提供任务级上下文；一个明确任务最好对应一个 thread。
  • thread -> 任务分解：多个 thread 不能替代清楚的任务分解，边界重叠时仍然可能互相干扰。
  • 安全边界 -> 审批：审批请求不是普通弹窗，而是越界或高风险动作交回给人判断。
  • 沙盒 -> review：沙盒限制影响范围，但不能证明结果正确；最终仍要看 diff、运行结果和风险。
  • 安全边界 -> 工作流：第一次任务要小，是为了先建立验收习惯，再进入真实项目。
  • 扩展能力 -> 上下文：Skills、plugins 和 MCP 的价值，是把更多项目规则、工具和外部能力接入任务现场。
