Reference 0009

Codex 提交与 PR 清单

验收通过不是终点。把改动交付出去时，要让未来的自己和 reviewer 看得懂、查得到、回得去。

一句话

stage 是选择，commit 是存档，PR 是申请，merge 是合入。

提交前检查

  • 先 `git status` 看有哪些文件变化。
  • 再看 diff，确认只提交通过验收的相关改动。
  • 不要把临时文件、密钥、`.env`、无关重构、全局样式、依赖升级混进同一个提交。
  • 如果一个 diff 包含多件事，优先拆成多个 commit 或先回退无关部分。
  • commit message 要描述真实范围，不要写成泛泛的 `update`。

常见 commit 类型

类型 意思 例子
feat 新增用户可见能力。 `feat: add reset link`
fix 修复错误行为。 `fix: restore redirect`
docs 只改文档。 `docs: update setup`
refactor 不改变行为的结构调整。 `refactor: simplify state`
test 新增或修正测试。 `test: cover login flow`

PR 描述五件事

  • 做了什么：用 2-4 条列出主要改动。
  • 为什么做：关联需求、bug、issue 或课程目标。
  • 怎么验证：列出测试、构建、lint、手动验证结果。
  • 有什么风险：说明影响范围、不确定点和需要人工确认的地方。
  • 重点看哪里：引导 reviewer 看关键文件、关键流程或高风险 hunk。

可复制提示词

请进入提交与 PR 准备阶段，不要继续新增功能。请基于当前 diff 和测试结果输出：建议 stage 哪些文件、一个不夸大范围的 commit message、PR 描述、风险说明、reviewer 需要重点看的地方，以及是否还有不应该提交的改动。
